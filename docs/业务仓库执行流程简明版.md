# 业务仓库执行流程简明版

## 背景

这份文档用于在业务仓库里执行 AI Coding Harness 日常流程。目标不是一次覆盖所有复杂场景，而是按 1、2、3、4 完成一次最小闭环。

## 研究问题

- 业务仓库已经接入 Harness 后，下一步怎么操作？
- 清空上下文后，如何让 AI 继续按规范执行？
- 每一步应该看什么结果，失败时停在哪里排查？

## 结论摘要

新业务仓库首次初始化：

```bash
bash /path/to/ai-coding-standards/scripts/init_business_repo.sh <repo-url-of-ai-coding-standards>
```

初始化后，一键执行新需求：

```bash
./ai run <spec-id> <source-doc>
```

分步排查时按下面 4 步执行：

1. 确认业务仓库状态
2. 准备一个需求文档
3. 生成并检查 spec
4. 执行任务并准备 PR

接入初期优先选择一个低风险的小需求，不要选 DB migration、权限、计费、风控类任务。

## 方案对比

### 方案 A：一键执行

适合快速验证主链路。

```bash
./ai run <spec-id> <source-doc>
```

优点：最快。

限制：失败时需要回头拆分定位。

示例：

```bash
./ai run demo-feature docs/requirements/demo-feature.md
```

### 方案 B：分步执行

适合接入初期和排查。

```bash
./ai status
./ai spec <spec-id> <source-doc>
./ai next <spec-id>
./ai work <spec-id> <task-id>
./ai pr <spec-id> <task-id> main
```

优点：每一步都能看到结果。

推荐接入初期使用方案 B。

### 方案 C：其他 AI 程序执行

适合使用 Claude Code、OpenClaw、Gemini CLI、Copilot CLI 或内部 AI 程序执行。

不要让这些工具执行 `./ai run`、`./ai spec`、`./ai work`、`./ai pr`，这些是 Codex 调度入口。请让它们读取：

```text
.ai-harness/.ai-standards/templates/prompts/serial-task-pipeline.md
```

然后提供集中输入：

```text
Spec ID: <spec-id>
Source document: <source-doc 或留空>
Target branch: <target-branch 或当前所在分支>
Execution Mode: portable-managed
```

也可以用脚本生成已替换占位符的提示词：

```bash
bash .ai-harness/.ai-standards/scripts/prepare_serial_task_prompt.sh <spec-id> [source-doc] [target-branch]
```

`target-branch` 可省略，默认使用当前所在 git 分支。`source-doc` 是普通业务需求文档，可以是 PRD、方案说明、缺陷描述、验收口径或用户故事；如果 spec 已存在，也可以省略 `source-doc` 继续执行。

该模式下 AI 程序可以按 `03-tasks.md` 依赖顺序串行执行 task，但每个 `task-id` 必须独立完成验证、acceptance 记录和 PR 材料，并在继续下一个 task 前暂停等待确认。

## 风险与限制

- `./ai work` 会进入开发动作，执行前应确认当前工作区没有不想混入的改动。
- `./ai pr` 会整理提交和推送材料，执行前应确认测试证据、风险、回滚方案已经明确。
- 如果任务复杂度是 `M` 或 `L`，不要直接开做，应先补齐 `02-design.md` 或拆小任务。

## 推荐做法

接入初期使用分步执行。下面假设：

- 业务仓库根目录是当前目录
- 需求文档是 `docs/requirements/demo-feature.md`
- spec id 是 `demo-feature`
- 首个 task id 是 `T1`

## 落地步骤

### 1. 初始化并确认业务仓库状态

新业务仓库首次接入时，在业务仓库根目录执行：

```bash
bash /path/to/ai-coding-standards/scripts/init_business_repo.sh <repo-url-of-ai-coding-standards>
```

如果已经接入过，直接执行：

```bash
./ai status
```

你应该看到：

```text
repo: ...
branch: ...
remote: ...
standards: .ai-harness/.ai-standards
dirty workspace:
specs:
```

如果提示找不到 `.ai-harness/.ai-standards`，说明业务仓库还没接入 Harness。先执行：

```bash
bash /path/to/ai-coding-standards/scripts/init_business_repo.sh <repo-url-of-ai-coding-standards>
```

### 2. 准备一个需求文档

创建或选择一个需求文档：

```bash
mkdir -p docs/requirements
```

示例路径：

```text
docs/requirements/demo-feature.md
```

需求文档至少写清楚：

- 要解决什么问题
- 用户怎么触发
- 期望结果是什么
- 不做什么
- 怎么验收

### 3. 生成并检查 spec

让 AI 基于需求文档生成并补齐 spec 四件套：

```bash
./ai spec demo-feature docs/requirements/demo-feature.md
```

这一步会创建或更新：

```text
.ai-harness/specs/demo-feature/01-requirements.md
.ai-harness/specs/demo-feature/02-design.md
.ai-harness/specs/demo-feature/03-tasks.md
.ai-harness/specs/demo-feature/04-acceptance.md
```

查看下一个可执行任务：

```bash
./ai next demo-feature
```

你应该看到类似：

```text
next task: T1
```

如果没有 next task，检查这些文件是否完整：

```text
.ai-harness/specs/demo-feature/01-requirements.md
.ai-harness/specs/demo-feature/02-design.md
.ai-harness/specs/demo-feature/03-tasks.md
.ai-harness/specs/demo-feature/04-acceptance.md
```

### 4. 执行任务并准备 PR

执行单个任务：

```bash
./ai work demo-feature T1
```

任务完成后准备 PR 材料：

```bash
./ai pr demo-feature T1 main
```

如果脚本要求人工确认，先阅读输出的 diff、测试证据、风险、回滚方案。确认无误后，可以使用：

```bash
AI_CONFIRM=1 ./ai pr demo-feature T1 main
```

如果使用一键命令：

```bash
./ai run demo-feature docs/requirements/demo-feature.md
```

失败后优先查看：

```text
.ai-harness/specs/demo-feature/03-tasks.md
```

文件底部的 `Run Checkpoint` 会给出恢复命令，例如：

```text
Resume Command: `./ai work demo-feature T1`
```

## 清空上下文后的启动提示词

新开 AI 会话后，可以直接复制这段：

```text
请按以下优先级读取并遵循规范：
1) 当前会话约束
2) .ai-harness/AGENTS.md
3) .ai-harness/.ai-standards/AGENTS.md
4) .ai-harness/.ai-standards/standards/*.md

我要继续执行业务仓库 AI Coding Harness 流程。
请先运行 ./ai status，确认当前仓库状态。
然后按分步流程执行：
1. ./ai spec <spec-id> <source-doc>
2. ./ai next <spec-id>
3. ./ai work <spec-id> <task-id>
4. ./ai pr <spec-id> <task-id> main

每一步执行后请说明：
- 执行了什么命令
- 结果是否通过
- 下一步是什么
- 如果失败，失败原因和需要我确认的事项
```

如果新开的不是 Codex，而是其他具备仓库读写能力的 AI 程序，可以复制这段：

```text
请读取并使用 .ai-harness/.ai-standards/templates/prompts/serial-task-pipeline.md。

模板输入：
- Spec ID: <spec-id>
- Source document: <source-doc 或留空>
- Target branch: <target-branch 或当前所在分支>

Execution Mode 使用 portable-managed。
不要执行 ./ai run、./ai spec、./ai work、./ai pr。
请先完成 Startup Checks，然后按当前 spec 的 03-tasks.md 依赖顺序串行执行 task。
每完成一个 task-id 后暂停，输出验证证据、风险、回滚方案、PR 材料和 next ready task，等待我确认是否继续。
```

## 验收标准

一次执行链路通过需要满足：

- `./ai status` 可以正常输出仓库和规范源信息
- `.ai-harness/specs/<spec-id>/01-requirements.md` 存在
- `.ai-harness/specs/<spec-id>/03-tasks.md` 存在，且至少有一个 `Task ID`
- `./ai next <spec-id>` 能找到任务，或明确说明为什么没有可执行任务
- `./ai work <spec-id> <task-id>` 完成一个任务
- `./ai pr <spec-id> <task-id> main` 输出 PR 所需材料
